## HTTP与HTTPS基本知识

### 1.URL和URI

URI，是uniform resource identifier，统一资源标识符。用来唯一的标识一个资源。
URL是uniform resource locator，统一资源定位器，它是一种具体的URI，即URL可以用来标识一个资源，而且还指明了如何locate这个资源。

如上，URI是一种语义上的抽象概念，可以是绝对的，也可以是相对的，而URL则必须提供足够的信息来定位，所以，是绝对的，而通常说的relative URL，则是针对另一个absolute URL，本质上还是绝对的。

比如：
    
    URL - http://example.com/some/page.html
    URI - /some/page.html

### 2.长连接和短连接

#### 1.OSI七层模型

物理层: 物理连网媒介
数据链路层：切割数据，寻找物理地址，纠错
**网络层：网络地址翻译成物理地址，并决定发送线路(IP协议、ICMP协议、ARP协议)**
**传输层：传输协议(TCP协议、UDP协议)**
会话层和表示层：通信管理，数据加解密管理等等
**应用层：提供网络服务(TCP协议包括FTP、HTTP、TELNET、SMTP等协议；UDP协议包括DNS、TFTP等协议)**

	关于Socket的理解，Socket是在应用层和传输层之间的一个抽象，它把TCP/IP层复杂的操作抽象为几个简单的接口供应用层调用以实现进程在网络中通信。简单来说，Socket是一组接口，把复杂的TCP/IP协议族隐藏在Socket接口后面，对用户来说，一组简单的接口就是全部，让Socket去组织数据，以符合指定的协议。所以无论应用层怎么玩，玩的基本对象就是它，而FTP、HTTP协议等都是建立在TCP协议之上的协议，相当于FTP协议规定一种socket的一套玩法、HTTP协议规定了另一种socket的一套玩法。
	
	因此我们始终都在操作Socket，而协议只是规范。
        
那么**短连接和长连接**其实都是针对Socket在某次请求结束要不要断开而言的。

更多关于通信协议的基础知识可参看[这里](http://blog.jobbole.com/105402/)

#### 2.短连接和长连接

首先，HTTP仅仅是个协议。它规定在这个协议下，socket应当如何通信。

HTTP1.0规定，一次请求只能由一方发起，处理完成就会立即断掉连接，如果继续通信就必须重新经历Socket的三次握手建立新的TCP连接，进行下一次请求。所谓的短连接就是这样，它无法在同一次连接中进行多次请求。

HTTP1.1新增了一个keep-alive，在一个TCP连接上可以经历多个HTTP请求和响应且响应完成之后不会断掉该连接，在下一次请求的过程中会复用当前连接。因此，它的主要作用是复用当前连接。
![长轮询](http://images2015.cnblogs.com/blog/150046/201602/150046-20160201230609475-1315946340.png)

HTTP2.0在1.1的基础上，支持了合并请求。对比如下：
![](http://img.blog.csdn.net/20160307004534945?watermark/2/text/aHR0cDovL2Jsb2cuY3Nkbi5uZXQv/font/5a6L5L2T/fontsize/400/fill/I0JBQkFCMA==/dissolve/70/gravity/Center)

可以说，HTTP已经支持长连接了，特别是到2.0。但均不是理想的即时通信协议，因为HTTP协议是被动的，服务端没有办法主动发起请求而只能被动的接受并处理消息。而最新的websocket协议尽管是基于HTTP协议的，但几乎是另外一套协议了。

所以要想真正实现即时通信，还是必须要换协议，比如xmpp，mqtt等。

### 3.HTTPS
HTTPS比HTTP多了个S，即是指的在原有HTTP基础上，增加了对数据进行保护的规定即，SSL及其继任者TLS。
#### 1 加密与解密基础
* 可逆加密和非可逆加密
    加密分为**可逆和非可逆**，而可逆加密根据其加密解密是否使用的同一个密钥而可以分为**对称加密和非对称加密**，对称加密即是指在加密和解密时使用的是同一个密钥，即**公钥和私钥**。公钥用于加密，所有人都可见，私钥用于解密，只有解密者持有。就算在一次加密过程中原文和密文发生泄漏，破解者在知道原文、密文和公钥的情况下无法推理出私钥，很大程度上保证了数据的安全性。

* 几个重要的使用相对频繁的加密算法：
    BASE64 - 不是加密，而是一种编码格式。
    
    MD5,SHA - 属于不可逆加密，使用它们进行加密不需要密钥，严格来讲，它们的本质是hash算法，与编码格式也是类似的不算加密，只不过其hash的过程非常复杂和繁琐，也就是说，一旦加密，没有办法从其加密结果得到原文。它们主要用于验证数据完整性，比如注册时用户名和密码经过不可逆加密生成了固定长度字符串p1，然后只将p传递给服务端，服务端拿到后保存；当下一次用户登录依然将用户名和密码经过不可逆加密生成了固定长度字符串p2，然后只将p2传递给服务端，服务端拿到p2并与上次保存的p1进行对比，如果一致登录成功。
    
    AES,DES - 属于可逆加密，且是对称加密。也就是说，密钥只有一个，通过这个密钥，可以对原文进行加密，并且可以解密。
    
    RSA,DSA - 属于可逆加密，且是非对称加密。它们的密钥有两个，一个公钥一个私钥。公钥是指公开的，谁都可以获取到的密钥，私钥则是指只有自己知道的的密钥，而公钥只负责加密，私钥只负责解密。
    
    加盐 - 加盐是指的在对原文进行上述几种方法加密之前，先混入一个带有身份标识的值。比如客户端对密码加密的盐值通常是手机的IEM码。加盐当然会让攻击难度加大，但其本质目的是缩小攻击范围，因为对于攻击者而言，拿到用户的加密结果以及加密算法，还得得到用户的IEM码，关键在于他要攻击多少人，就得破解多少个用户的IEM码，其攻击成本会上升很多。

#### 2.HTTPS加密过程

>区别：
一、HTTPS协议需要到CA申请证书，一般免费证书很少，需要交费。 
二、HTTP是超文本传输协议，信息是明文传输，HTTPS 则是具有安全性的SSL加密传输协议。 
三、HTTP和HTTPS使用的是完全不同的连接方式，用的端口也不一样，前者是80，后者是443。 
四、HTTP的连接很简单，是无状态的；HTTPS协议是由SSL+HTTP协议构建的可进行加密传输、身份认证的网络协议，比HTTP协议安全。

简单来说，HTTP存在的最大问题就是明文传输，明文传输导致的两个弊端是**双方身份无法认证**和**数据传输不安全**。那么SSL解决该问题的基本流程如下：
<pre>
1 身份认证阶段
client -> server : SSL协议的版本号，加密算法的种类等等信息。
server -> client :SSL协议的版本号，加密算法的种类等等信息，以及自己的证书（该证书内有公钥，因为所有客户端都可以取到这把钥匙，所以叫公钥，只是个名称）。
client -> server : 拿到了服务端的证书，开始验证服务端合法性，主要是看证书内容和实际情况是否吻合，比如证书域名和实际域名比较，或者用公钥来解证书内的数字签名。验证通过后，就可以进行数据传输了。

2 数据传输阶段
client -> server : 生成一个对称密钥，对请求的内容（如url及其参数）进行加密，然后用证书内的公钥对这个对称密钥再加一次密，一起发送到服务端。
server -> client : 服务端用自己的私匙对拿到的加过密的对称密钥解密，然后用这把对称密钥对请求内容进行解密。再根据请求内容执行相关的操作，请求执行完毕返回时，再次用对称密钥对返回内容加密。
client -> server : 拿到服务端返回的内容时，只需要用对称密钥对其解密就可以了。
</pre>

## RESTful

RESTful是一种设计思想，该思想的基本点是基于一切事物皆资源对象，只要通过正确的URI就可以得到唯一的资源，或者说，URI唯一标识着该资源。

* GET /tickets 
获取ticket的集合
* GET /tickets/12
获取某个具体的ticket
* DELETE /tickets/12
删除id为12的ticekt 
* PUT /tickets/12
更新id为12的ticekt 

### 1.使用名词而不是动词

`/tickets` 和 `/getAllTickets`
之所以推荐前者，主要原因是HTTP已经有针对Action的描述，**GET ,POST,PUT,DELETE**，已经基本能够涵盖大部分类型的请求。

### 2.资源之间的关系描述
GET `/cars/711/drivers/4`
仅看该uri基本就能明白，该接口是通过查询cars集合中名为711的car，然后通过该car找到id为4的driver并返回。反过来，如果不用这种方式该如何为这个接口命名？

### 3.接口版本
GET `https://developer.github.com/v3/media/#request-specific-version`

这里重点说版本，在设计和构建API时，必须确保新API对于现有的API不会造成破坏，那么必须对其进行区分。API的版本控制策略通常有3种模式：

* 1 无版本。即平台的API永远只有一个版本，所有的用户都必须使用最新的API，任何API的修改都会影响到平台所有的用户，甚至平台的整个生态系统。
* 2 点对点。即平台的API版本自带版本号，用户根据自己的需求选择使用对应的API，需要使用新的API特性，用户必须自己升级。
* 3 兼容性版本控制。平台只有一个版本，但是最新版本需要兼容以前版本的API行为。

最理想的方式当然是第三种，但成本比较高且并不容易完全实现，比如：
```java
/**
 * @deprecated Use {@link #setButton(int, CharSequence, Message)} with
 *             {@link DialogInterface#BUTTON_POSITIVE}.
 */
 @Deprecated
 public void setButton(CharSequence text, Message msg) {
    setButton(BUTTON_POSITIVE, text, msg);
 }

 /**
  * @deprecated Use {@link #setButton(int, CharSequence, Message)} with
  *             {@link DialogInterface#BUTTON_NEGATIVE}.
  */
 @Deprecated
 public void setButton2(CharSequence text, Message msg) {
    setButton(BUTTON_NEGATIVE, text, msg);
 }
```
上面是通过重载的方式实现兼容的一种方式。所以简单来说，兼容旧接口的意思，就是在客户端入参不变和请求方式不变的情况下，接口被修改但依然可以得到正确的结果。这时，客户端在版本升级时就可以选择非强制升级，因为旧版接口依然可以用；相反，如果无法兼容旧接口，如果以老方式调用会报错，这时客户端在版本升级时就必须强制升级。而且多数时候，强制与非强制都是由接口兼容决定的。新写接口还是修改原接口，亦或者兼容旧接口并非是技术问题，而是经验问题，需要结合实际场景和需求预测。

当然，将版本号直接放进uri并非是REST风格，因为它与业务资源没有任何关系，另一种做法是，将版本号放在HTTP头信息中，但不如放入URL方便和直观。

### 4.过滤信息

### 5.HTTP状态码和自定义状态码
HTTP状态码如下：
>200 OK - [GET]：服务器成功返回用户请求的数据，该操作是幂等的（Idempotent）。
201 CREATED - [POST/PUT/PATCH]：用户新建或修改数据成功。
202 Accepted - [*]：表示一个请求已经进入后台排队（异步任务）
204 NO CONTENT - [DELETE]：用户删除数据成功。
304 not modified - HTTP缓存有效。
400 INVALID REQUEST - [POST/PUT/PATCH]：用户发出的请求有错误，服务器没有进行新建或修改数据的操作，该操作是幂等的。
401 Unauthorized - [*]：表示用户没有权限（令牌、用户名、密码错误）。
403 Forbidden - [*] 表示用户得到授权（与401错误相对），但是访问是被禁止的。
404 NOT FOUND - [*]：用户发出的请求针对的是不存在的记录，服务器没有进行操作，该操作是幂等的。
406 Not Acceptable - [GET]：用户请求的格式不可得（比如用户请求JSON格式，但是只有XML格式）。
410 Gone -[GET]：用户请求的资源被永久删除，且不会再得到的。
422 Unprocesable entity - [POST/PUT/PATCH] 当创建一个对象时，发生一个验证错误。
500 INTERNAL SERVER ERROR - [*]：服务器发生错误，用户将无法判断发出的请求是否成功。

HTTP状态码定义了很多响应状态，尽管并不能满足多数复杂的需求。而状态码对于开发人员来说其作用主要是用来跟踪错误。错误分为很多种，断网，无权限，请求错误，处理错误等等。而此类错误都囊括在HTTP状态码中了，无须我们来定义。但还有很多错误是HTTP协议无法定义的，比如支付失败，三方登陆失败，购物车已满添加商品失败等等，此类错误在于它们是允许出错的，但我们依然需要作出区分，比如支付失败是因为参数错误，还是程序错误，或者是余额不足，甚至银行在规定时间段禁止交易呢？这些错误就需要自己定义，方便跟踪或者提示。

对于HTTP状态错误，api原则上是不需要返回结构体的，因为这类错误一般是程序上的错误，当拿到500的时候，就说明服务端报错，客户端拿到它就够了，当拿到404时也是如此。
对于自定义状态错误，就必须要返回结构体了，返回结构体才能分析是何原因导致的错误。

结构体举例：
```java
public class BaseModel {

  /**
   * 登录已失效状态码
   */
  public static final int ERR_TOKEN_EMPTY = 25;
  public static final int ERR_TOKEN_TIMEOUT = 26;
  public static final int ERR_TOKEN = 27;
  public static final int ERR_TOKEN_INVALID = 28;

  /**
   * 当且仅当响应正确才返回true，反之返回false。
   */
  public boolean isSuccess() {
    return success;
  }

  /**
   * 获得返回码
   */
  public int getStatusCode() {
    return code;
  }

  /**
   * 当且仅当失效才返回true，反之返回false。
   */
  public boolean isInvalidCookie() {
    return ERR_TOKEN_EMPTY == code
        || ERR_TOKEN_TIMEOUT == code
        || ERR_TOKEN == code
        || ERR_TOKEN_INVALID == code;
  }

  public String message;
  public String extmsg;
  public int code;
  public boolean success;
}
```
结构体是客户端与服务端交互的基本容器，它一般包含message，code，requestId，data四个部分。

比如，当login失败时，message就负责装载错误的具体信息，code就代表此类错误标识。

* 1 如果没有message字段的返回，就无法知道错误的具体原因，那么当用户点击登录按钮就只会得到一个统一的解释“登录失败”。至于为何失败不得而知。
* 2 如果没有code返回，那么我们可以拿到HTTP状态码判读200并提示message内的字段，比如密码错误，账户名错误，或校验码错误。看上去很完美，可是当token失效，客户端需要作重新登录，该如何判断呢。
* 3 服务为每一个请求唯一生成一个请求ID，当客户端开发者无法自助解决问题时，可以联络服务开发者，同时提供该请求ID。一个好的服务，服务开发者应当可以根据此ID，定位到该次请求的所有相关信息，包括入参，接口名，响应错误等等信息进而定位问题，快速解决问题。
* 4 当接口请求成功，data就可以作为其载体正常返回数据。而data是载体这一点需要充分体现，任何类型的任何数据都应当作为它的成员，而它本身是一个抽象的概念，没有具体实际的意义，如果data以其一种具体的形式出现则会在意义上造成混淆。
    比如如下结构：
```

    }
}
```
可以很清楚的看到，data这个容器里包含了一个有有关token的成员变量，且token的值是一长串的字符串，而如果返回结构如下：

3.HTTPS

HTTPS比HTTP多了个S，即是指的在原有HTTP基础上，增加了对数据进行保护的规定即，SSL及其继任者TLS。

1 加密与解密基础

  • 可逆加密和非可逆加密
    加密分为可逆和非可逆，而可逆加密根据其加密解密是否使用的同一个密钥而可以分为对称加密和非对称加密，对称加密即是指在加密和解密时使用的是同一个密钥，即公钥和私钥。公钥用于加密，所有人都可见，私钥用于解密，只有解密者持有。就算在一次加密过程中原文和密文发生泄漏，破解者在知道原文、密文和公钥的情况下无法推理出私钥，很大程度上保证了数据的安全性。

  • 几个重要的使用相对频繁的加密算法：
    BASE64 - 不是加密，而是一种编码格式。

    MD5,SHA - 属于不可逆加密，使用它们进行加密不需要密钥，严格来讲，它们的本质是hash算法，与编码格式也是类似的不算加密，只不过其hash的过程非常复杂和繁琐，也就是说，一旦加密，没有办法从其加密结果得到原文。它们主要用于验证数据完整性，比如注册时用户名和密码经过不可逆加密生成了固定长度字符串p1，然后只将p传递给服务端，服务端拿到后保存；当下一次用户登录依然将用户名和密码经过不可逆加密生成了固定长度字符串p2，然后只将p2传递给服务端，服务端拿到p2并与上次保存的p1进行对比，如果一致登录成功。

    AES,DES - 属于可逆加密，且是对称加密。也就是说，密钥只有一个，通过这个密钥，可以对原文进行加密，并且可以解密。

    RSA,DSA - 属于可逆加密，且是非对称加密。它们的密钥有两个，一个公钥一个私钥。公钥是指公开的，谁都可以获取到的密钥，私钥则是指只有自己知道的的密钥，而公钥只负责加密，私钥只负责解密。

    加盐 - 加盐是指的在对原文进行上述几种方法加密之前，先混入一个带有身份标识的值。比如客户端对密码加密的盐值通常是手机的IEM码。加盐当然会让攻击难度加大，但其本质目的是缩小攻击范围，因为对于攻击者而言，拿到用户的加密结果以及加密算法，还得得到用户的IEM码，关键在于他要攻击多少人，就得破解多少个用户的IEM码，其攻击成本会上升很多。

2.HTTPS加密过程

区别：
一、HTTPS协议需要到CA申请证书，一般免费证书很少，需要交费。
二、HTTP是超文本传输协议，信息是明文传输，HTTPS 则是具有安全性的SSL加密传输协议。
三、HTTP和HTTPS使用的是完全不同的连接方式，用的端口也不一样，前者是80，后者是443。
四、HTTP的连接很简单，是无状态的；HTTPS协议是由SSL+HTTP协议构建的可进行加密传输、身份认证的网络协议，比HTTP协议安全。

简单来说，HTTP存在的最大问题就是明文传输，明文传输导致的两个弊端是双方身份无法认证和数据传输不安全。那么SSL解决该问题的基本流程如下：

1 身份认证阶段
client -> server : SSL协议的版本号，加密算法的种类等等信息。
server -> client :SSL协议的版本号，加密算法的种类等等信息，以及自己的证书（该证书内有公钥，因为所有客户端都可以取到这把钥匙，所以叫公钥，只是个名称）。
client -> server : 拿到了服务端的证书，开始验证服务端合法性，主要是看证书内容和实际情况是否吻合，比如证书域名和实际域名比较，或者用公钥来解证书内的数字签名。验证通过后，就可以进行数据传输了。

2 数据传输阶段
client -> server : 生成一个对称密钥，对请求的内容（如url及其参数）进行加密，然后用证书内的公钥对这个对称密钥再加一次密，一起发送到服务端。
server -> client : 服务端用自己的私匙对拿到的加过密的对称密钥解密，然后用这把对称密钥对请求内容进行解密。再根据请求内容执行相关的操作，请求执行完毕返回时，再次用对称密钥对返回内容加密。
client -> server : 拿到服务端返回的内容时，只需要用对称密钥对其解密就可以了。

3.接口版本

GET https://developer.github.com/v3/media/#request-specific-version

这里重点说版本，在设计和构建API时，必须确保新API对于现有的API不会造成破坏，那么必须对其进行区分。API的版本控制策略通常有3种模式：

  • 1 无版本。即平台的API永远只有一个版本，所有的用户都必须使用最新的API，任何API的修改都会影响到平台所有的用户，甚至平台的整个生态系统。
  • 2 点对点。即平台的API版本自带版本号，用户根据自己的需求选择使用对应的API，需要使用新的API特性，用户必须自己升级。
  • 3 兼容性版本控制。平台只有一个版本，但是最新版本需要兼容以前版本的API行为。

最理想的方式当然是第三种，但成本比较高且并不容易完全实现，比如：

1./**
2. * @deprecated Use {@link #setButton(int, CharSequence, Message)} with
3. *             {@link DialogInterface#BUTTON_POSITIVE}.

上面是通过重载的方式实现兼容的一种方式。所以简单来说，兼容旧接口的意思，就是在客户端入参不变和请求方式不变的情况下，接口被修改但依然可以得到正确的结果。这时，客户端在版本升级时就可以选择非强制升级，因为旧版接口依然可以用；相反，如果无法兼容旧接口，如果以老方式调用会报错，这时客户端在版本升级时就必须强制升级。而且多数时候，强制与非强制都是由接口兼容决定的。新写接口还是修改原接口，亦或者兼容旧接口并非是技术问题，而是经验问题，需要结合实际场景和需求预测。

当然，将版本号直接放进uri并非是REST风格，因为它与业务资源没有任何关系，另一种做法是，将版本号放在HTTP头信息中，但不如放入URL方便和直观。

5.HTTP状态码和自定义状态码

HTTP状态码如下：

200 OK - [GET]：服务器成功返回用户请求的数据，该操作是幂等的（Idempotent）。
201 CREATED - [POST/PUT/PATCH]：用户新建或修改数据成功。
202 Accepted - [*]：表示一个请求已经进入后台排队（异步任务）
204 NO CONTENT - [DELETE]：用户删除数据成功。
304 not modified - HTTP缓存有效。
400 INVALID REQUEST - [POST/PUT/PATCH]：用户发出的请求有错误，服务器没有进行新建或修改数据的操作，该操作是幂等的。
401 Unauthorized - [*]：表示用户没有权限（令牌、用户名、密码错误）。
403 Forbidden - [*] 表示用户得到授权（与401错误相对），但是访问是被禁止的。
404 NOT FOUND - [*]：用户发出的请求针对的是不存在的记录，服务器没有进行操作，该操作是幂等的。
406 Not Acceptable - [GET]：用户请求的格式不可得（比如用户请求JSON格式，但是只有XML格式）。
410 Gone -[GET]：用户请求的资源被永久删除，且不会再得到的。
422 Unprocesable entity - [POST/PUT/PATCH] 当创建一个对象时，发生一个验证错误。
500 INTERNAL SERVER ERROR - [*]：服务器发生错误，用户将无法判断发出的请求是否成功。

HTTP状态码定义了很多响应状态，尽管并不能满足多数复杂的需求。而状态码对于开发人员来说其作用主要是用来跟踪错误。错误分为很多种，断网，无权限，请求错误，处理错误等等。而此类错误都囊括在HTTP状态码中了，无须我们来定义。但还有很多错误是HTTP协议无法定义的，比如支付失败，三方登陆失败，购物车已满添加商品失败等等，此类错误在于它们是允许出错的，但我们依然需要作出区分，比如支付失败是因为参数错误，还是程序错误，或者是余额不足，甚至银行在规定时间段禁止交易呢？这些错误就需要自己定义，方便跟踪或者提示。

对于HTTP状态错误，api原则上是不需要返回结构体的，因为这类错误一般是程序上的错误，当拿到500的时候，就说明服务端报错，客户端拿到它就够了，当拿到404时也是如此。
对于自定义状态错误，就必须要返回结构体了，返回结构体才能分析是何原因导致的错误。

结构体举例：

结构体是客户端与服务端交互的基本容器，它一般包含message，code，requestId，data四个部分。

比如，当login失败时，message就负责装载错误的具体信息，code就代表此类错误标识。

  • 1 如果没有message字段的返回，就无法知道错误的具体原因，那么当用户点击登录按钮就只会得到一个统一的解释“登录失败”。至于为何失败不得而知。
  • 2 如果没有code返回，那么我们可以拿到HTTP状态码判读200并提示message内的字段，比如密码错误，账户名错误，或校验码错误。看上去很完美，可是当token失效，客户端需要作重新登录，该如何判断呢。
  • 3 服务为每一个请求唯一生成一个请求ID，当客户端开发者无法自助解决问题时，可以联络服务开发者，同时提供该请求ID。一个好的服务，服务开发者应当可以根据此ID，定位到该次请求的所有相关信息，包括入参，接口名，响应错误等等信息进而定位问题，快速解决问题。
  • 4 当接口请求成功，data就可以作为其载体正常返回数据。而data是载体这一点需要充分体现，任何类型的任何数据都应当作为它的成员，而它本身是一个抽象的概念，没有具体实际的意义，如果data以其一种具体的形式出现则会在意义上造成混淆。
    比如如下结构：
1.{
2.    "message":"xxx",
3.    "code":200,
4.    "data":{
5.        "token":"djhsdioie138478302jdw"
6.    }
7.}

可以很清楚的看到，data这个容器里包含了一个有有关token的成员变量，且token的值是一长串的字符串，而如果返回结构如下：

1.{
2.    "message":"xxx",
3.    "code":200,
4.    "data":"djhsdioie138478302jdw"
5.}

那么data这一串字符串的意义难以知晓。

命名

命名是一门学问，它的主要作用是让一年后的自己及其他开发人员能基本看懂，并且顺藤摸瓜，特别体现在阅读源码的时候，一份高质量源码能够让你在少量的注释下快速掌握该段代码所实现的基本功能或实现意图。

1.单复数

比如GET http://xxx/tickets和http://xxx/ticket

前者一看tickets这个单词是复数就知道这是获取一个集合，后者则是会让人摸不着头脑，这是要获取什么。因为在第一眼看到前者的uri之后，第一我们不光知道这是获取一个集合的操作，第二我们还会知道或许有另一个获取某一项的接口即http://xxx/tickets/12以及第三我们还能猜想应该会有DELETE、PUT http://xxx/tickets/12的操作支持，而这仅仅发生在第一眼看到的那一瞬间。而后者是做不到的。

在代码编写过程中也是如此，java.net.URLClassLoader内方法：
